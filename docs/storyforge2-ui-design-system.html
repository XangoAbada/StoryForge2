StoryForge2 / design system

Mapa wszystkich głównych elementów UI w jednym roboczym atlasie.

Ten plik porządkuje aktualny interfejs jako system: tokeny, komponenty bazowe, wzorce AI, relacje planu, modale, edytor scen, eksport i ekrany robocze. Każdy element można oznaczyć notatką redesignu w prawym panelu.

Diagnoza

Co powinien trzymać profesjonalny design system StoryForge2

Aplikacja ma już spójny charakter: spokojny warsztat pisarski, panele robocze, dużo relacji i mocny wzorzec AI przy każdym polu. Design system powinien zabezpieczać powtarzalność tych zachowań, nie tylko kolory.

01
Tokeny decyzji

Kolory, typografia, promienie, cienie, focus, gęstość i wartości statusów muszą być nazwane semantycznie: akcja, AI, relacja, ostrzeżenie, gotowość.

Komponenty z pełnymi stanami

Każdy przycisk, chip, pole, karta i modal powinien mieć stany: domyślny, hover, focus, aktywny, disabled, loading, error i empty.

Wzorce domenowe

Najważniejsze są wzorce specyficzne dla aplikacji: AI przy polu, panel kontekstu promptu, propozycja/akceptacja, relacje `+/-`, kanban planu, edytor scen.

Reguły dostępności

System powinien wymagać `aria-label`, `role="dialog"`, `aria-modal`, zamykania Escape, czytelnych tytułów relacji i widocznych stanów focus.

Treść UI po polsku

Wszystkie etykiety, opisy, statusy, błędy i prompt labels powinny używać poprawnych polskich znaków oraz jednego tonu: rzeczowo, spokojnie, bez nadmiaru instrukcji.

Dokumentowanie zmian

Każdy redesign powinien zapisywać problem, miejsce w aplikacji, proponowany kierunek, priorytet i kryteria akceptacji. Panel po prawej tworzy taki zapis.

03

Przyciski akcji

`primary-button`, `secondary-button`, `ghost-button`, `icon-button`

Używane w formularzach, modalach, dashboardzie, planie, scenach, świecie i eksporcie.

Statusy i komunikaty

`status-pill`, `autosave-status`, `warning-text`, `success-text`
Gotowy W kolejce Wymaga uwagi Błąd

Zapisano automatycznie - 10:42

System powinien jasno odróżniać gotowość, ostrzeżenie, błąd i stan neutralny.

Pola formularza z AI

`field-shell`, `ai-field-button`, `ai-context-add-button`

To najważniejszy wzorzec produktu: generowanie dotyczy tylko pola obok przycisku i przechodzi przez akceptację propozycji.

Pola wyboru i przełączniki

`select`, `range`, `toggle-row`, `segmented-control`

Segmenty służą do małych zestawów opcji, select do list, range do reasoning effort, checkboxy do list rozdziałów i źródeł kontekstu.

Chipy, relacje i tagi

`chapter-chip`, `world-relation-chip`, `relation-chip`
Wątek: zdrada Beat: midpoint Postać: Mira

Chipy relacji odpinają wyłącznie relację. Nie usuwają encji kanonicznej: beatu, wątku, rozdziału, sceny, postaci ani elementu świata.

Nawigacja etapów i zakładki

`concept-stage-tab`, `plan-stage-step`, `character-tab`, `world-tab`

Etapy pokazują proces pracy, zakładki przełączają widoki jednej encji lub panelu.

Notatki

Jak pracować z notatkami redesignu

Kliknij `Notatka` przy komponencie, opisz problem i wygeneruj zapis. Notatki są zapisywane lokalnie w przeglądarce, więc możesz wracać do tego pliku i eksportować listę do Markdown albo JSON.

06
1. Wybierz element

Przycisk przy komponencie wypełnia nazwę elementu i źródło. Możesz też wpisać własny element ręcznie w panelu.

2. Nazwij problem

Najlepsza notatka mówi, co boli: spacing, proporcje, hierarchia, stan disabled, copy, dostępność albo model interakcji.

3. Dodaj kryteria akceptacji

Zapisuj konkret: co ma być sprawdzalne po redesignie, na jakich ekranach i przy jakich stanach komponentu.
